## Image Format Comparison

### PNG (Recommended for Static Screenshots)

**Pros:**
- Lossless compression maintains perfect quality
- Excellent for screenshots with text (sharp rendering)
- Universal browser/software compatibility
- Transparency support with alpha channels
- Predictable rendering across all devices

**Cons:**
- Larger file sizes (can be several MB for detailed images)
- Not suitable for animations

**Best For:** Terminal screenshots with text, UI menus, static displays

**Technical Note:** PNG is ideal for complex graphics and screenshots where text clarity is paramount. The lossless compression algorithm finds patterns without discarding any details.

**Sources:**
- [SVG Genie - Image Format Guide 2025](https://www.svggenie.com/blog/svg-vs-png-vs-jpg-guide)
- [Kate Scott - Image File Formats Guide](https://katescott.co/blog/jpg-vs-png-vs-gif-vs-svg-image-file-formats)

### GIF (For Simple Animations)

**Pros:**
- Animation support (standard for web)
- Wide compatibility
- Small file sizes for simple graphics

**Cons:**
- Limited to 256 colors (can look degraded)
- Binary transparency (jagged edges)
- Lossy compression can blur text
- Large file sizes for high-quality recordings

**Best For:** Short, simple terminal animations (5-15 seconds)

**Sources:**
- [PageCloud - PNG vs JPG vs GIF vs SVG](https://www.pagecloud.com/blog/web-images-png-vs-jpg-vs-gif-vs-svg)
- [Modo Agency - Image Format Guide](https://modomodoagency.com/blog/2019/01/jpg-vs-png-vs-gif-vs-svg-image-format-guide/)

### SVG (For Vector-Based Recordings)

**Pros:**
- Infinitely scalable without quality loss
- Text remains selectable/searchable
- Small file sizes (code-based, not pixels)
- Can be edited in text editors
- Modifiable by JavaScript or CSS

**Cons:**
- Not ideal for complex raster images
- Some older browsers have limited support
- Requires special tools for terminal recordings

**Best For:** Terminal recordings generated by tools like `termtosvg`, logos, icons

**Technical Note:** SVG files are actually code—mathematical equations defining shapes, colors, and size. This is why they retain quality at any scale.

**Sources:**
- [Elementor - SVG vs PNG](https://elementor.com/blog/svg-vs-png-what-are-the-differences/)
- [SitePoint - GIF, PNG, JPG or SVG](https://www.sitepoint.com/gif-png-jpg-which-one-to-use/)

### Format Recommendation Matrix

```
┌────────────────────┬─────────────┬──────────────────────────┐
│ Use Case           │ Format      │ Rationale                │
├────────────────────┼─────────────┼──────────────────────────┤
│ Static menu        │ PNG         │ Text clarity, lossless   │
│ Short demo (5-10s) │ GIF         │ Animation, compatibility │
│ Long demo (30s+)   │ SVG or MP4  │ Quality, file size       │
│ Code-like output   │ SVG         │ Searchable, scalable     │
│ Tmux split panes   │ PNG         │ Layout precision         │
│ Interactive flow   │ GIF or SVG  │ Shows user interaction   │
└────────────────────┴─────────────┴──────────────────────────┘
```

## Recording Tools & Techniques

### VHS by Charmbracelet (⭐ HIGHLY RECOMMENDED for 2025-2026)

**VHS** is the modern standard for creating terminal GIFs and videos from code. It's become the industry favorite for CLI tool demonstrations.

**Key Features:**
- Write terminal GIFs as code using `.tape` files
- Generate GIFs, MP4s, WebM videos, or PNG frame sequences
- Records in custom terminal but honors your bindings
- Can publish to `vhs.charm.sh` for hosted GIFs
- Supports CI/CD integration with GitHub Actions
- Can be used for integration testing

**Installation:**
```bash
# macOS or Linux
brew install vhs

# Arch Linux
pacman -S vhs

# Nix
nix-env -iA nixpkgs.vhs

# Windows
scoop install vhs

# Docker
docker run --rm -v $PWD:/vhs ghcr.io/charmbracelet/vhs <cassette>.tape
```

**Requirements:** ttyd and ffmpeg

**Quick Start:**
```bash
# Record your terminal activity
vhs record > demo.tape

# Or write a tape file manually
cat > demo.tape <<'EOF'
Output demo.gif
Set FontSize 18
Set Width 1200
Set Height 600

Type "ls -la"
Enter
Sleep 2s

Type "cd src"
Enter
Sleep 1s
EOF

# Generate the GIF
vhs demo.tape

# Publish to hosted URL
vhs --publish demo.tape
```

**Output Formats:**
```tape
Output out.gif        # Animated GIF
Output out.mp4        # MP4 video
Output out.webm       # WebM video
Output frames/        # PNG frame sequence
```

### Asciinema (Classic Terminal Recorder)

**Best For:** Recording terminal sessions with playback on asciinema.org

**Key Features:**
- Records right where you work—in your terminal
- Output is text-based (can copy-paste from recordings)
- Lightweight file sizes
- Can embed recordings on websites

**Quick Start:**
```bash
# Install
brew install asciinema  # macOS
apt install asciinema   # Ubuntu/Debian

# Record
asciinema rec demo.cast

# End recording
# Press Ctrl+D or type exit

# Play locally
asciinema play demo.cast

# Upload to asciinema.org
asciinema upload demo.cast
```

**Recording tmux Sessions with Asciinema:**

If you want to record your tmux session with all windows and splits:

1. Set up your tmux session:
   ```bash
   tmux new -s demo-session
   # Create windows, splits, start processes
   ```

2. Detach from session:
   ```bash
   # Press prefix+d
   ```

3. Record with asciinema:
   ```bash
   asciinema rec --command "tmux attach -t demo-session" tmux-$(date +%F--%H%M).cast
   ```

4. When finished, detach again (prefix+d)

**One-command detach+reattach** (if already inside tmux):
```bash
tmux detach -E 'asciinema rec -c "tmux attach" tmux.cast'
```

**Known Issue:** iTerm2 has a bug where it doesn't render frames correctly after tmux recordings. Use Apple's Terminal app instead, or exit and restart the shell.

**Sources:**
- [Asciinema Official Site](https://asciinema.org/)
- [GitHub - Recording tmux session](https://github.com/asciinema/asciinema/wiki/Recording-tmux-session)

## Optimal Dimensions & Sizing

### GitHub README Display Constraints

**Key Facts:**
- GitHub renders images at max-width of **~888-900 pixels**
- Default GitHub image preview is **~610 x 610 pixels**
- Copy-pasted images auto-resize to **150 x 150 pixels**
- Images scale responsively on mobile devices

**Sources:**
- [GitHub Gist - Markdown Insert Screenshot](https://github.com/cavaunpeu/markdown-insert-screenshot/blob/master/README.md)
- [w3schools - GitHub README Images](https://www.w3schools.io/file/github-readme-image/)

### Recommended Dimensions for Terminal Screenshots

#### Standard Terminal Window
```
Width:  800-1200 pixels (sweet spot: 1000-1100px)
Height: 400-800 pixels (depends on content)
Aspect: 16:9 or 2:1 preferred
Format: PNG for stills, GIF/SVG for animations
```

#### Hero Image / Banner
```
Width:  1200-1400 pixels (GitHub container width)
Height: 400-600 pixels
Aspect: 2:1 to 3:1 (wide banners work well)
Format: PNG or SVG
```

#### Mobile-Optimized
```
Width:  800 pixels minimum
Height: Auto (maintain aspect ratio)
Note:   Test at 375px viewport (mobile)
```

### File Size Guidelines

| Format | Target Size | Maximum |
|--------|-------------|---------|
| PNG    | < 500 KB    | < 2 MB  |
| GIF    | < 1 MB      | < 5 MB  |
| SVG    | < 200 KB    | < 1 MB  |
| MP4    | < 2 MB      | < 10 MB |

**Why:** Large images are a primary cause of slow-loading websites and READMEs.

**Sources:**
- [Kate Scott - Image File Formats](https://katescott.co/blog/jpg-vs-png-vs-gif-vs-svg-image-file-formats)
- [Amber Design - Best Image Formats](https://www.amberddesign.com/image-formats/)

### Controlling Image Size in Markdown

GitHub Markdown supports HTML image tags for precise sizing:

```html
<!-- Set width only (height auto-adjusts) -->
<img src="./docs/images/demo.png" width="800" alt="Demo">

<!-- Set both dimensions -->
<img src="./docs/images/demo.png" width="800" height="500" alt="Demo">

<!-- Responsive sizing with max-width -->
<img src="./docs/images/demo.png" style="max-width: 100%; height: auto;" alt="Demo">
```

**Best Practice:** Set width only and let height adjust to preserve aspect ratio.

**Sources:**
- [GitHub Gist - Resize Images](https://gist.github.com/stevecondylios/dcadb4fc73e63f27a3bbcf17e52058bf)
- [w3schools - GitHub README Images](https://www.w3schools.io/file/github-readme-image/)

#### Essential Elements:

1. **Show the tool in action** - Not just static output
2. **Include the command** - Show what user types
3. **Highlight key features** - Annotations if needed
4. **Use realistic data** - Not "foo bar baz"
5. **Keep it concise** - 10-15 seconds max for GIFs
6. **Clear typography** - Font size 14-18pt minimum
7. **Proper contrast** - Light text on dark, or vice versa

- Cluttered output
- Generic placeholder text
- No context for what's happening
- Too long (30+ seconds)

## Tmux-Specific Considerations

### Capturing Tmux Split Panes

#### Method 1: Terminal Emulator Screenshot
Most reliable for complex layouts:

- **iTerm2 (macOS):** `Cmd+Shift+S` or Edit → Save Selected Text
- **Kitty:** `kitty +kitten icat screenshot.png`
- **Alacritty:** Use external tool (scrot, maim, flameshot)
- **GNOME Terminal:** Built-in screenshot feature

#### Method 2: tmux capture-pane
Captures text content (good for simple cases):

```bash
# Capture current pane to file
tmux capture-pane -p > pane-output.txt

# Capture with ANSI color codes
tmux capture-pane -p -e > pane-output-color.txt

# Capture specific pane
tmux capture-pane -t 0 -p > pane0.txt

# Capture entire history
tmux capture-pane -p -S -3000 > full-history.txt
```

#### Method 3: External Screenshot Tools
For full window captures:

```bash
# scrot (X11)
scrot tmux-screenshot.png

# maim (X11)
maim -s tmux-screenshot.png

# ImageMagick import
import tmux-screenshot.png

# flameshot (cross-platform)
flameshot gui
```

#### Method 4: Convert to Image with aha
```bash
# Capture pane with colors, convert to HTML, then image
tmux capture-pane -p -e | aha --black > output.html
wkhtmltoimage output.html screenshot.png
```

### Tmux Recording Considerations

**Challenges:**
- Split panes can be hard to see in small recordings
- Status bar may contain personal info
- Window/pane borders may not be clear
- Color schemes matter more

**Solutions:**

1. **Simplify Layout for Demos:**
   ```bash
   # Create clean demo session
   tmux new-session -s demo
   tmux set -g status-right ""
   tmux set -g status-left ""
   ```

2. **Increase Font Size:**
   ```tape
   Set FontSize 18  # VHS
   ```

3. **Focus on One Pane at a Time:**
   - Zoom panes during recording: `prefix + z`
   - Or record each pane separately

4. **Use Clear Borders:**
   ```bash
   tmux set -g pane-border-style "fg=colour240"
   tmux set -g pane-active-border-style "fg=colour250"
   ```

5. **Demo-Friendly Status Bar:**
   ```bash
   tmux set -g status-left "[Demo]"
   tmux set -g status-right ""
   ```

### Recording Multi-Pane Tmux with Asciinema

**Best Practice:** Record the entire tmux session, not from within

```bash
# 1. Set up your multi-pane layout
tmux new-session -s recording

# Create your splits
tmux split-window -h
tmux split-window -v
tmux select-pane -t 0

# 2. Detach
tmux detach

# 3. Record the attachment
asciinema rec --command "tmux attach -t recording" tmux-demo.cast

# 4. When done, detach to stop recording (prefix + d)
```

**Warning:** Recording tmux can generate enormous files (450MB+ for extended sessions). Compress with `xz` to reduce to ~15MB.

**Alternative:** Use **tvmux** for more efficient tmux pane recording.

## Hero Image Guidelines

### What Makes an Effective Hero Image

**Hero Image = First Screenshot/GIF at Top of README**

#### Qualities of Great Hero Images:

1. **Instantly Communicates Value**
   - Shows the tool solving a problem
   - Demonstrates key unique feature
   - Makes user think "I need this!"

2. **Professionally Presented**
   - Clean terminal window
   - Good typography (18pt+ font)
   - Proper color contrast
   - Branded color scheme

3. **Quick to Understand**
   - 5-10 seconds max for GIFs
   - One clear action/flow
   - No ambiguous output

4. **Technically Excellent**
   - Sharp text rendering (PNG preferred)
   - Proper dimensions (1000-1200px wide)
   - Optimized file size (< 1MB)
   - Works on dark/light backgrounds

### Hero Image Types by Tool Category

| Tool Type | Best Hero Format | Example |
|-----------|------------------|---------|
| **TUI (Terminal UI)** | Static PNG | Dashboard, menu system |
| **CLI with prompts** | Animated GIF | Interactive workflow |
| **Syntax highlighter** | Before/after PNG | Side-by-side comparison |
| **Performance tool** | GIF with timing | Show speed improvement |
| **File manager** | Animated GIF | Navigation flow |
| **Build tool** | Static PNG + progress | Clean output display |

## Implementation Checklist

### Pre-Recording Preparation

```markdown
Setup Checklist:
□ Clean terminal history
□ Remove sensitive information from prompt/config
□ Set appropriate terminal size (120x30 or 100x30)
□ Choose readable font (14-18pt)
□ Select high-contrast color scheme
□ Clear status bars of personal info (tmux/vim)
□ Prepare example data/files
□ Test commands to avoid typos
□ Close unnecessary applications
□ Use incognito/demo mode if showing browser
```

### Recording Best Practices

```markdown
During Recording:
□ Type at normal speed (not too fast)
□ Use clear, realistic examples
□ Keep recordings under 15 seconds
□ Show the command before output
□ Include brief pauses (1-2s) for comprehension
□ Highlight key moments
□ Avoid showing errors unless intentional
□ End at a logical conclusion
```

### Post-Recording Optimization

```markdown
After Recording:
□ Compress images (ImageOptim, TinyPNG, etc.)
□ Test file size (target < 1MB for GIFs)
□ Verify text is readable at GitHub's max-width
□ Check on both dark and light backgrounds
□ Test mobile responsiveness
□ Add descriptive alt text
□ Use semantic file names (demo.gif, not screenshot1.gif)
□ Store in docs/images/ directory
□ Commit source files (.tape, .cast) for updates
```


## Key Takeaways

1. **Visual demos are crucial** - 42% more stars for repos with screenshots
2. **Use VHS for 2025-2026** - Modern, code-based terminal recording
3. **PNG for stills, GIF/SVG for animations** - Choose format based on use case
4. **1000-1200px width is optimal** - Works well on GitHub's layout
5. **Keep GIFs under 15 seconds** - Short, focused demos are most effective
6. **Hero image is make-or-break** - First impression determines engagement
7. **Optimize file sizes** - Target < 1MB for fast loading
